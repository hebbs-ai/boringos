Prerequisites & EC2 setup

Before SSH-ing in, make sure the instance and its Security Group are configured correctly.

Recommended instance type

Use caseInstanceRAMNotes
Development / testingt3.small2 GBTight but functional
Staging / light productiont3.medium4 GBRecommended minimum
Productiont3.large8 GBComfortable headroom

Security Group inbound rules

PortSourcePurpose
22Your IP onlySSH
30000.0.0.0/0BoringOS HTTP server
5432Do not openPostgres — stays internal to the instance
⚠️
Never expose port 5432 Postgres will be bound to 127.0.0.1 inside Docker. Adding a Security Group rule for 5432 would expose your database directly to the internet.

Connect and update

terminal 
# Ubuntu
ssh -i your-key.pem ubuntu@<ec2-public-ip>

# Amazon Linux
ssh -i your-key.pem ec2-user@<ec2-public-ip>
terminal 
# Ubuntu — update packages
sudo apt update && sudo apt upgrade -y

# Amazon Linux — update packages
sudo dnf update -y

Install Node 22

BoringOS requires Node 22 or newer. Install via nvm so you can manage versions easily.

terminal 
# Install nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash

# Reload shell so nvm is on PATH
source ~/.bashrc

# Install Node 22 and set it as default
nvm install 22
nvm use 22
nvm alias default 22

# Verify
node --version
ℹ️
Expected output node --version should print v22.x.x or higher.

Install Docker

Docker is used to run the Postgres database in an isolated, manageable container. Follow the steps for your OS.

Ubuntu 22.04

terminal — Ubuntu 
# Install prerequisites
sudo apt install -y ca-certificates curl gnupg
sudo install -m 0755 -d /etc/apt/keyrings

# Add Docker's official GPG key
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

# Add Docker repository
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install Docker
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io

# Enable and start Docker service
sudo systemctl enable --now docker

# Allow your user to run Docker without sudo
sudo usermod -aG docker $USER
newgrp docker

# Verify
docker --version
docker run --rm hello-world

Amazon Linux 2023

terminal — Amazon Linux 
sudo dnf install -y docker
sudo systemctl enable --now docker
sudo usermod -aG docker $USER
newgrp docker

# Verify
docker --version
docker run --rm hello-world
✅
Confirm Docker works docker run --rm hello-world should print "Hello from Docker!" If you see a permission error, log out and back in so the group change takes effect.

Postgres with persistent storage

Postgres runs as a Docker container. Its data is stored in a named Docker volume (boringos-pgdata) which lives on the EC2 instance's disk and persists independently of the container lifecycle.

Step 1 — Create the volume

terminal 
docker volume create boringos-pgdata

# See where Docker stores it on the host disk
docker volume inspect boringos-pgdata

Step 2 — Start the container

Replace YOUR_STRONG_PASSWORD with a real password before running. You will need it again in the next section.

terminal 
docker run -d \
  --name boringos-pg \
  --restart unless-stopped \
  -e POSTGRES_USER=boringos \
  -e POSTGRES_PASSWORD=YOUR_STRONG_PASSWORD \
  -e POSTGRES_DB=boringos \
  -v boringos-pgdata:/var/lib/postgresql/data \
  -p 127.0.0.1:5432:5432 \
  postgres:16
ℹ️
Why 127.0.0.1:5432 and not just 5432? Binding to 127.0.0.1 restricts connections to processes on this machine only. Binding to 0.0.0.0:5432 would expose Postgres to the public internet even if a Security Group rule were accidentally added.

Step 3 — Verify it is ready

terminal 
# Should print: /var/run/postgresql:5432 - accepting connections
docker exec boringos-pg pg_isready -U boringos

# Confirm container is Up
docker ps
✅
Expected: "accepting connections" If you see "no response" or an error, wait 5 seconds and retry. The container takes a moment to initialize on first boot.

Persistence summary

Action takenData survives?
docker restart boringos-pgYes
docker stop then docker startYes
docker rm boringos-pg then recreate with same -vYes
EC2 reboot (sudo reboot)Yes — Docker auto-restarts the container
docker volume rm boringos-pgdataNo — this deletes the volume
EC2 instance terminatedNo — root EBS volume is gone
⚠️
For true production durability Attach a separate EBS volume, mount it at /data, and use a bind mount (-v /data/postgres:/var/lib/postgresql/data) instead of a named volume. That way the data survives full EC2 replacement — just reattach the EBS volume to a new instance.

Clone and build the repo

terminal 
# Install git if not already present
sudo apt install -y git          # Ubuntu
sudo dnf install -y git          # Amazon Linux

# Clone
git clone <your-repo-url>
cd boringos

# Install all workspace dependencies
pnpm install

# Compile TypeScript for every package in the monorepo
pnpm -r build
ℹ️
Build time The first build takes 1–2 minutes. It compiles TypeScript for every package in the monorepo. Subsequent builds are faster because of incremental compilation.
7

Environment variables

Create an .env file in the quickstart directory. This keeps secrets out of your shell history.

terminal 
cd examples/quickstart

cat > .env << 'EOF'
DATABASE_URL=postgres://boringos:YOUR_STRONG_PASSWORD@127.0.0.1:5432/boringos
PG_EMBEDDED=false
NODE_ENV=production
PORT=3000
AUTH_SECRET=REPLACE_AFTER_RUNNING_COMMAND_BELOW
EOF

Generate a secure value for AUTH_SECRET:

terminal 
openssl rand -hex 32

Copy the output and paste it as the value for AUTH_SECRET in your .env file.

⚠️
Save AUTH_SECRET somewhere safe If the server restarts without this value, all existing login sessions are invalidated. Store it in AWS Secrets Manager or SSM Parameter Store for anything beyond a one-off setup.

Variable reference

VariableRequiredDescription
DATABASE_URLYesPostgres connection string
PG_EMBEDDEDYesfalse forces external Postgres; true forces embedded; unset = auto
NODE_ENVRecommendedSet to production
PORTNoHTTP port, default 3000
AUTH_SECRETRecommendedSigns session tokens; random on each boot if not set
RESEND_API_KEYNoEnables email notifications
8

Start the server

terminal 
cd examples/quickstart

# Load .env into the current shell session
set -a && source .env && set +a

pnpm start

On first boot, BoringOS connects to Postgres and runs all schema migrations automatically. All tables are created with IF NOT EXISTS guards — safe to run repeatedly.

expected output 
BoringOS running at http://localhost:3000
Health: http://localhost:3000/health

Confirm from another terminal (or from your local machine using the public IP):

terminal 
# From the EC2 instance
curl http://localhost:3000/health

# From your local machine
curl http://<ec2-public-ip>:3000/health
✅
Expecting {"status":"ok"} Any other response: check server logs for errors, confirm Postgres is ready with docker exec boringos-pg pg_isready -U boringos, and verify PG_EMBEDDED=false and DATABASE_URL are both set correctly.

Keep it running with PM2

PM2 is a process manager that keeps BoringOS alive after you disconnect from SSH, restarts it automatically on crashes, and hooks into the system startup so it comes back after a reboot.

Install PM2

terminal 
npm install -g pm2

Create an ecosystem config

This tells PM2 where the app lives and which .env file to load.

examples/quickstart/ecosystem.config.cjs 
module.exports = {
  apps: [{
    name: "boringos",
    script: "pnpm",
    args: "start",
    cwd: "/home/ubuntu/boringos/examples/quickstart",
    env_file: ".env",
    restart_delay: 3000,
    max_restarts: 10,
    log_date_format: "YYYY-MM-DD HH:mm:ss",
  }]
};
ℹ️
Update cwd if your username is not ubuntu Amazon Linux uses ec2-user, so the path would be /home/ec2-user/boringos/...

Start and configure auto-restart on reboot

terminal 
cd examples/quickstart

# Start the app
pm2 start ecosystem.config.cjs

# Save the current process list
pm2 save

# Generate the startup script
pm2 startup
# Copy and run the command it prints (starts with "sudo env PATH=...")
⚠️
Run the command PM2 prints pm2 startup prints a sudo env PATH=... command. You must copy and run that exact command — it registers the PM2 daemon with systemd so it starts on reboot.

Useful commands

CommandWhat it does
pm2 logs boringosStream live application logs
pm2 statusShow all running processes and their state
pm2 restart boringosRestart the server (picks up new code after a rebuild)
pm2 stop boringosStop without removing from PM2 list
pm2 delete boringosRemove completely from PM2
pm2 monitLive dashboard with CPU and memory usage

Test a reboot

terminal 
sudo reboot

After reconnecting, verify everything came back:

terminal 
docker ps                                         # boringos-pg should be Up
pm2 status                                        # boringos should be online
curl http://localhost:3000/health                  # should return {"status":"ok"}

Final checklist

  • EC2 Security Group: port 3000 open, port 22 from your IP only, port 5432 closed
  • Node 22 installed — node --version prints v22.x.x
  • pnpm 9 installed — pnpm --version prints 9.x.x
  • Docker running — docker ps works without sudo
  • Postgres container is Up — docker ps shows boringos-pg
  • Postgres volume exists — docker volume inspect boringos-pgdata succeeds
  • pg_isready returns "accepting connections"
  • PG_EMBEDDED=false is set in .env
  • DATABASE_URL uses your real password, not the placeholder
  • AUTH_SECRET is set and saved somewhere safe (SSM / Secrets Manager)
  • curl http://localhost:3000/health returns {"status":"ok"}
  • PM2 startup hook registered — pm2 startup command was run
  • Reboot test passed: server and Postgres both came back automatically
