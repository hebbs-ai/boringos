1. The Big Picture

📱
Think of BoringOS as Android
The Framework = Android OS. It handles the heavy lifting: authentication, permissions, storage, app lifecycle.

Connector SDK = Android SDK libraries. Like how Android gives you CameraManager and LocationManager, we give you GmailClient and CalendarClient. Pure libraries. No OS internals exposed.

Modules = Android Apps. Each app (module) is a self-contained lego block. Instagram and WhatsApp both use the Camera SDK, but they build completely different experiences. Similarly, a Productivity Module and an EA Module both use GmailClient, but their tools do completely different things.

Permissions = Runtime, not install-time. Just like Android asks for camera permission only when you tap the camera button, our modules check scopes only when they actually need them.
🔌
Connectors are Power Adapters
You travel to different countries (Google, Slack, GitHub, Microsoft). Each country has different power sockets (OAuth endpoints, token formats, scope systems). A connector is the universal adapter -- same shape on the module's side, different prongs on the provider's side.

Every connector implements one interface: ConnectorDefinition. Learn the shape once, plug into anything.

3. What the Peer Review Found

The review identified 15 issues. Three are high severity (correctness/architecture), one is a sequencing priority, the rest are refinements.

ISSUE #2
Token refresh regression
The bug: Spec deletes runWithRefresh but hands clients a frozen token: string. If a Gmail batch runs for 90 seconds and the token expires mid-flight, it silently fails. No retry, no refresh.

Analogy: Like giving a driver a gas card that expires in 1 hour, but no way to refuel. Works for short trips, breaks on long ones.
ISSUE #3
Blocking requestScopes jams the queue
The bug: requestScopes() blocks for up to 5 minutes waiting on a human click. The default agent queue is serial (concurrency = 1). One blocked tool = every other agent waiting.

Analogy: Like a single-lane drive-through where one car parks at the window to make a phone call. Everyone behind is stuck.
ISSUE #4
accountId pollutes every tool
The bug: Every tool requires accountId: z.string() in its schema. But the module_connector_bindings table already resolves accounts server-side. The agent has to know something it shouldn't.

Analogy: Like requiring every restaurant customer to specify which power outlet their table uses. The building's wiring handles that.
ISSUE #14
Encryption should ship first
The gap: OAuth tokens are stored as plaintext JSONB in the database today. This is a live security gap that depends on zero redesign work. Bundling it with the SDK redesign delays the fix unnecessarily.
ISSUE #5
Manifest vs runtime authority unclear
Module declares connectors in manifest AND checks scopes at runtime. Which one actually governs access? Never stated explicitly.
ISSUE #7 + #8
Skills + tool names not addressed
Every module ships skill files that teach agents tool names. Changing client APIs without updating skills breaks live agents. Agent-facing tool names must be explicitly preserved.
ISSUE #12 + #13
Two specs in one, all breaks at once
Safe SDK work (typed clients) is bundled with risky redesign (multi-account, scope upgrades). All breaking changes land simultaneously. Should be phased so safe work ships without waiting on unproven work.
ISSUE #15
"kind" name collision
Module.kind = "connector" | "module" | "hybrid". ConnectorDefinition.kind = "google" | "slack". Same field name, different meanings. Minor but confusing.

4. How the Design Evolved (v1 to v2)

The core vision is unchanged. Universal connector interface, typed clients, modules as lego blocks, Android-style permissions. What changed is the mechanics -- how tokens refresh, how scopes upgrade, how accounts resolve, and in what order we ship.

5. The Five Key Fixes

Fix 1: Token Provider, not Static String

v1 -- Broken
Proposal // Token is a frozen string const token = await deps.getConnectorToken( "google", accountId, "my-module" ); // Client gets a snapshot -- if it expires // mid-flight, the call fails silently const gmail = new GmailClient(token.accessToken); // 90 seconds later... 401. No retry. await gmail.listMessages(); // FAILS
→
v2 -- Fixed
Revised // Token is a provider function const conn = await deps.getConnectorToken( "google", "my-module" ); // Client calls getToken() on each request // and retries on 401 with a fresh token const gmail = new GmailClient(conn.getToken); // 90 seconds later... auto-refresh. await gmail.listMessages(); // WORKS
⛽
Analogy: Gas card vs. fuel account
v1 gave you a pre-paid gas card with a fixed balance. If the trip was longer than expected, you're stranded.
v2 gives you a fuel account that auto-refills. The car checks the balance before each fill-up and tops up if needed. Same destination, no stranding.

Fix 2: Approval Flow, not Blocking Promise

v1 -- Jams the queue
Proposal // Tool BLOCKS for up to 5 minutes // waiting for user to click consent const result = await deps.requestScopes( "google", accountId, missingScopes ); // Meanwhile, the serial queue is frozen. // Every other agent waits. // // Queue: [BLOCKED] [waiting] [waiting]
→
v2 -- Non-blocking
Revised // Tool returns immediately with error return { ok: false, error: "needs_scope", result: { missing, consentUrl } }; // Agent creates approval task (existing HITL) // User grants scope in browser // Agent rewakes, retries -- scope now granted // // Queue: [done] [running] [running]
🏦
Analogy: Bank teller vs. ATM + callback
v1 was like a bank teller who freezes the entire line to call your spouse for wire transfer approval. Everyone waits.
v2 is like the teller saying "you need approval -- here's a form, come back when it's signed." Next customer, please. When you return with the signed form, you go right back to the window.

Fix 3: Server Resolves the Account

v1 -- Agent must know
Proposal // Every tool requires accountId inputs: z.object({ accountId: z.string(), maxResults: z.number(), }) // Agent has to pick which account // Pollutes every tool schema deps.getConnectorToken( "google", input.accountId, "my-mod" );
→
v2 -- Server resolves
Revised // No accountId in tool inputs inputs: z.object({ maxResults: z.number(), }) // Auth manager resolves from binding: // (tenantId, moduleId, provider) -> accountId deps.getConnectorToken( "google", "my-mod" );
🏨
Analogy: Hotel room keys
v1 was like a hotel where every time you use the elevator, you have to type your room number. Why? The hotel already knows which room you're in from your keycard.
v2 is like tapping your keycard -- the system resolves your room, floor, and access rights automatically. The user configured the binding (checked in), the system handles the rest.

Fix 4: "provider" not "kind"

v1 -- Confusing
Proposal // Module.kind = "connector" | "module" // ConnectorDefinition.kind = "google" // Same name, different meaning! Module { kind: "connector" } ConnectorDef { kind: "google" }
→
v2 -- Clear
Revised // Module.kind = what type of module // ConnectorDef.provider = which service // No collision Module { kind: "connector" } ConnectorDef { provider: "google" }

Fix 5: Ship Safe Work First

v1 -- Big bang
Proposal (4 phases, all risky) Phase 1: Schema + Auth Manager Phase 2: Connector Packages Phase 3: Built-in Modules Phase 4: Multi-Account UI All breaking changes land together Safe work blocked by risky work Encryption bundled with redesign
→
v2 -- Incremental
Revised (4 phases, de-risked) Phase 0: Encryption (standalone) Phase 1: Typed clients (additive) Phase 2: Auth manager + migration Phase 3: Multi-account UI (if needed) Safe work ships independently Deprecation bridge for executeAction Security gap closed immediately

6. New Phased Timeline

Phase 0 No Risk
Credential Encryption
  • Encrypt credentials JSONB at rest (current connectors table)
  • Add BORINGOS_ENCRYPTION_KEY env var
  • Migration script for existing plaintext rows
  • Zero API changes, zero module impact
Phase 1 Low Risk
ConnectorDefinition + Typed Clients
  • Add types to @boringos/module-sdk (additive)
  • Add typed methods to GmailClient, CalendarClient, SlackClient
  • Keep executeAction as deprecated (backward compatible)
  • Add ConnectorDefinition exports to connector packages
  • Update skill files (gmail.md, calendar.md)
  • Agent-facing tool names unchanged
Phase 2 Medium Risk
Auth Manager + Schema Migration
  • New tables: connector_accounts, connector_oauth_apps, module_connector_bindings
  • Build AuthManager with binding-based account resolution
  • Token-provider function replaces runWithRefresh
  • Built-in modules migrate to deps.getConnectorToken
  • Remove deprecated executeAction
  • Remove hardcoded OAUTH_PROVIDERS, loadGoogleCreds, etc.
Phase 3 Only If Needed
Multi-Account UI + Scope UX
  • Shell UI for multiple accounts per connector
  • Module settings for account bindings
  • Scope consent integrated with approval flow
  • Ships only when real demand exists
